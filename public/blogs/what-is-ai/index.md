这篇文章用一个“搭建和管理项目”的目录来讲 AI。这样会更接近真实使用场景：AI 不是孤零零的聊天窗口，而是一套可以理解目标、调用工具、按流程完成任务的协作系统。

## 0. Fork 项目

Fork 原本是 GitHub 里的概念，意思是把别人的项目复制一份到自己的账号下，再基于这份副本修改。

理解 AI 也可以从这里开始：AI 并不是凭空产生答案，它会基于已有知识、上下文和你的输入继续生成。你可以把一次对话理解成“Fork 一个临时工作空间”：你给它背景，它在这个背景上继续推理；你给它文件，它就基于文件内容继续改；你给它目标，它就围绕目标组织下一步。

这里有几个基础名词：

- AI：人工智能，是让机器表现出理解、生成、判断、执行等能力的一类技术。
- 模型：AI 的“大脑”，比如大语言模型负责理解文字、写代码、总结内容。
- Prompt：你给 AI 的任务说明，包含目标、背景、规则和输出要求。
- 上下文：AI 当前能看到的信息，比如你的问题、项目文件、错误日志、截图。

所以，Fork 项目这一节的核心是：先给 AI 一个可工作的上下文。上下文越清楚，它越像是在你的项目里工作，而不是凭空猜。

## 1. 安装

“安装”可以理解为把 AI 能力接进你的工作环境。

普通聊天 AI 只负责回答问题；接入工具后的 AI 才能真正做事。比如它可以读取文件、运行命令、打开网页、调用 GitHub、部署到 Vercel 或 Cloudflare。这些能力不是模型天然拥有的，而是系统把工具授权给它之后，它才能使用。

这就像你安装一个开发环境：

- 模型负责理解和推理。
- 工具负责执行真实操作。
- 权限决定它能碰哪些东西。
- 日志和反馈告诉它操作是否成功。

如果没有工具，AI 只能告诉你“应该怎么做”。有了工具，它才能帮你“真的做完”。

## 2. 部署

部署就是把能力放到真实环境里运行。

对于 AI 来说，部署不是只有上线模型这么简单。更常见的部署方式是把 AI 放进一个工作流里，让它能围绕目标持续执行。

例如你让 AI 搭建博客，它需要做的不只是写几句话，而是：

1. 理解博客模板结构。
2. 修改配置和文章文件。
3. 提交到 GitHub。
4. 部署到 Vercel。
5. 打开线上页面验证结果。

这时 AI 就不只是“回答者”，而更像“执行者”。这也是 Agent 概念出现的原因。

Agent 可以理解成会围绕目标行动的 AI。它会拆任务、调用工具、观察结果、修正错误，直到任务完成。

## 3. 创建 Github App 链接仓库

这里可以用 GitHub App 来理解“Agent 为什么需要权限”。

如果一个 Agent 要帮你改文章、删文章、部署项目，它必须能访问你的仓库。但这个访问不能是无限制的，否则风险太高。比较好的方式是给它一个明确授权：只能访问指定仓库，只能做指定范围内的操作。

这就是权限边界。

在 AI 系统里，权限边界非常重要。因为模型会生成行动建议，工具会执行真实操作。如果工具权限过大，错误操作的影响也会变大。

### 创建密钥

密钥就是身份凭证。

在你的博客系统里，GitHub App 的 `.pem` 私钥相当于管理员钥匙。导入这把钥匙后，网页后台才能代表 GitHub App 去创建 token，然后向仓库提交文章、修改配置或删除文件。

这也解释了为什么不是每个人都能改你的博客：页面虽然公开，但真正保存需要私钥。没有私钥，就不能提交到你的仓库。

AI 使用工具时也有类似逻辑：

- 没有密钥，就只能看公开信息。
- 有只读密钥，就只能读取。
- 有写入密钥，才能修改。
- 密钥权限越大，越需要小心保存。

所以密钥不能发给别人，不能截图内容，不能提交到公开仓库。

### 环境变量

环境变量是给程序读取的配置。

比如部署网站时，你不会把密钥直接写进代码，而是放到 Vercel、Cloudflare 或本地 `.env` 里。程序运行时从环境变量读取它们。

在 AI 项目里，环境变量常见用途包括：

- API Key：调用模型或第三方服务。
- 仓库信息：GitHub owner、repo、branch。
- 存储配置：R2 bucket、S3 endpoint、访问密钥。
- 安全配置：加密 key、访问密码。

环境变量的好处是代码可以公开，敏感信息不公开。对于 Agent 来说，这也是安全边界的一部分。

## 4. 完成

一个任务什么时候算完成？

不是 AI 说“完成了”就完成了，而是结果被验证过。

比如发布文章，至少要确认：

- GitHub 仓库里出现了新提交。
- Vercel 部署成功。
- `/blog` 列表能看到文章。
- 文章详情页能打开。
- 旧文章确实删除或隐藏。

这就是 Agent 工作里非常重要的一环：验证。

一个可靠的 Agent 不应该只会执行，还应该会检查执行结果。如果失败，它要能根据错误继续排查，而不是假装成功。

## 5. 删除

删除是最需要谨慎的操作之一。

AI 可以帮你删除文章、删除文件、清理旧配置，但删除前必须明确范围。比如“删除其他文章”到底是只从列表隐藏，还是从仓库里彻底删除？这两种结果不一样。

这也是为什么真正有用的 AI 协作，需要人来做最终决策。

AI 擅长：

- 整理信息。
- 生成内容。
- 执行重复步骤。
- 排查常见错误。
- 把复杂任务拆成流程。

人需要负责：

- 决定目标。
- 判断风险。
- 确认删除范围。
- 保管密钥。
- 决定哪些内容可以公开。

AI 不是替你承担责任，而是帮你降低完成事情的成本。

## 6. 配置

Skill 就可以放在“配置”这个位置理解。

Skill 不是模型本身，而是一套做事说明。它告诉 Agent：遇到某类任务时应该怎么做、用什么工具、注意什么风险、如何验证。

比如：

- 部署 Cloudflare 前，要检查登录状态、绑定和构建目录。
- 调试 bug 时，要先找根因，再改代码。
- 操作 GitHub 时，不要提交密钥。
- 做前端页面时，要检查移动端和文字溢出。

如果模型是大脑，工具是手脚，那么 Skill 就像工作手册。它让 Agent 不只是“聪明”，还更稳定、更有流程感。

### 6.1 网站配置

拿你这个博客来说，网站配置包括：

- 站点标题、用户名、头像。
- 社交按钮，比如 GitHub、LINUX DO、邮箱。
- 首页卡片布局。
- 文章索引和分类。
- 图床地址和图片资源。

AI 可以帮你改这些配置，也可以帮你写文章、部署、验证。但真正让它稳定工作的，是模型、工具、权限、环境变量、Skill 和验证流程一起配合。

所以一句话总结：

AI 是能力，模型是大脑，工具是手脚，Skill 是方法，Agent 是能按目标持续执行的人机协作者。
